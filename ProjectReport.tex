% ---------- Abstract ----------
\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

Modern web applications face increasingly sophisticated threats from automated bots,
credential stuffing, and volumetric attacks. Cloud providers market CDNs, WAFs and
bot mitigation services as comprehensive defences, yet there is a gap in independent,
isolated empirical evaluation of these claims, particularly on non-enterprise tiers.

This project deploys a cloud-hosted honeypot disguised as a legitimate streaming
service to attract real-world malicious traffic, comparing Cloudflare (free-tier) and
BunnyCDN (Advanced Shield) against identical attack scenarios. Testing revealed that
while both platforms effectively absorbed volumetric network-layer attacks, HTTP/2
protocol-compliant floods bypassed Cloudflare's edge in a significant proportion of
cases. Rate limiting enforced thresholds but permitted indefinite attack cycling, and
telemetry gaps on lower tiers limited incident visibility throughout.

To complement CDN testing, an unsupervised machine learning pipeline using HDBSCAN
clustering was trained on natural datasets accumulated from real-world logs and
deployed to classify live honeypot traffic into behavioural attack profiles,
identifying patterns ranging from PHP webshell deployment and path traversal probing
to secrets harvesting, IoT device exploitation, and Exchange / RDP reconnaissance. An agentic LLM layer
interprets these clusters and augments its context through RAG, auto-generating
structured incident tickets, creating a lightweight closed-loop threat detection
system.

The findings demonstrate a gap between vendor security claims across different tiers
of service and real-world effectiveness, and propose a practical, evidence-based
approach to continuous threat monitoring.

\section{Project Description}

One way in which defenders can learn an attacker's strategy is to dupe them into
attacking a honeypot. The attacker believes they are attacking a production system,
but in actual fact it is a fake entity designed to trap the attacker. As the attacker
attempts to compromise the honeypot, we can learn about their attack strategy: what
vectors they are using, what commands they are executing and in what order. Learning
the attack motivation and strategy enables cybersecurity experts to harden their
defences. If an attacker realises they are interacting with a honeypot they will
immediately stop; building a system that emulates a real system convincingly is an
open challenge. The goal is to keep them interacting while also learning their
strategies.

% ---------- Useful Definitions ----------
\section{Useful Definitions}
\label{sec:definitions}

\begin{description}[leftmargin=2cm, style=nextline]

  \item[Honeypot]
    A deliberately vulnerable system designed to attract and trap attackers. It mimics
    a legitimate production environment to deceive malicious actors, allowing defenders
    to observe attack strategies and techniques without risking real systems.

  \item[Content Delivery Network (CDN)]
    A geographically distributed network of servers that cache and deliver web content
    from locations closer to end users. Modern CDNs also provide DDoS protection, bot
    mitigation, and web application firewalls.

  \item[Web Application Firewall (WAF)]
    A security solution that filters, monitors, and blocks HTTP/HTTPS traffic to and
    from web applications based on predefined security rules. WAFs protect against SQL
    injection, XSS, and other OWASP Top 10 vulnerabilities.

  \item[CAPTCHA]
    A challenge-response test designed to determine whether a user is human or an
    automated bot. Common implementations include image-selection puzzles, text
    recognition, or invisible background checks (reCAPTCHA v3).

  \item[Bot]
    An automated software application that performs repetitive tasks over the internet.
    Bots can be benign (search engine crawlers) or malicious (credential stuffing
    tools, web scrapers, vulnerability scanners).

  \item[DoS (Denial of Service)]
    An attack that attempts to make a machine, network, or service unavailable to its
    intended users by overwhelming it with illegitimate requests, exhausting resources
    such as CPU, memory, and bandwidth~\cite{stallings2003dos}.

  \item[DDoS (Distributed Denial of Service)]
    A cyberattack that attempts to overwhelm a target system, service, or network with
    excessive traffic from multiple sources, rendering it unavailable to legitimate
    users.

  \item[Adaptive Deception Technology]
    Security systems that dynamically adjust their behaviour and responses based on
    attacker interactions, making honeypots more realistic and harder to detect while
    gathering intelligence on attack strategies.

  \item[Serverless Computing]
    A cloud computing model where the cloud provider manages server infrastructure,
    automatically allocating resources as needed. Developers deploy code without
    managing underlying servers, paying only for actual compute time consumed.

  \item[API (Application Programming Interface)]
    A set of protocols and tools that allows different software applications to
    communicate with each other. In web contexts, APIs enable front-end applications
    to interact with back-end services through defined endpoints.

  \item[Rate Limiting]
    A security control that restricts the number of requests a user or IP address can
    make to a service within a specified time period, helping prevent abuse, scraping,
    and denial-of-service attacks.

  \item[Telemetry]
    The automated collection and transmission of data from remote sources for
    monitoring and analysis. In security contexts, telemetry includes logs, metrics,
    and event data used to understand system behaviour and detect threats.

  \item[Challenge Page]
    An intermediate page presented to users before granting access to a website,
    typically containing a CAPTCHA or JavaScript verification to distinguish humans
    from automated bots.

  \item[Brute Force Attack]
    A hacking method that involves using trial and error to crack passwords or gain
    unauthorised access~\cite{fortinet-brute-force}.

\end{description}

% ---------- Introduction ----------
\section{Introduction}

Modern web applications face increasingly sophisticated threats ranging from automated
bots to web scrapers and malicious actors constantly probing for vulnerabilities.
Cloud service providers have responded by offering integrated security solutions such
as Content Delivery Networks (CDNs) with built-in protection features, Web Application
Firewalls (WAFs), and bot mitigation services. Companies like Cloudflare, Google Cloud
and Amazon Web Services market these as comprehensive defence mechanisms capable of
identifying and blocking malicious traffic before it reaches production systems.

However, there may be a significant difference between marketing promises and
real-world effectiveness. While these providers claim robust bot detection, DDoS
mitigation and attack prevention, there is limited independent research evaluating
whether these features deliver on their promises when facing genuine threats. Do
CAPTCHAs effectively stop modern bots? Can CDN-based scraping protection actually
prevent data harvesting? Are the various security toggles meaningfully different, or
largely cosmetic?

This project aims to address these questions through empirical analysis using adaptive
deception technology. By deploying a cloud-based honeypot disguised as a legitimate
streaming service (\texttt{hdtvstreams.com}), we create a controlled environment that
attracts real malicious traffic while capturing detailed telemetry data.

% ---------- Motivation ----------
\section{Motivation}

The motivation for this research stems from several observations about the current
state of cloud security:

\begin{description}[leftmargin=2.5cm, style=nextline]

  \item[Market-Reality Gap]
    Cloud providers invest heavily in marketing their security features, but
    independent verification of these claims is scarce. There are no clear SLAs
    specifying how effective these security offerings should be.

  \item[The CAPTCHA Paradox]
    End users are increasingly burdened with CAPTCHA challenges, sometimes multiple
    times per session. It is not fully verified whether CAPTCHAs are security theatre
    that inconveniences legitimate users while barely impacting malicious automation.

  \item[Configuration Complexity]
    Modern CDNs like Cloudflare offer dozens of toggleable security settings. However,
    it is unclear which combination actually enhances security versus which are
    redundant. This research aims to isolate and test individual features.

  \item[Multi-Cloud Comparisons]
    Organisations often employ multiple cloud providers, yet there is limited
    comparative data on how their security offerings stack up against identical
    threats. Does Cloudflare's bot detection outperform AWS WAF? Is Google Cloud CDN's
    scraping protection more or less effective?

  \item[Evolution of Attack Patterns]
    CDN attack-intelligence models may be trained on older patterns; this project
    captures contemporary attack patterns against realistic web applications.

\end{description}

% ---------- Research Questions ----------
\section{Research Questions}

\begin{enumerate}

  \item \textbf{How effective are CDN-provided security mechanisms at detecting and
    mitigating real-world malicious traffic when deployed on non-enterprise tiers?}

  \item \textbf{How does the effectiveness of security features differ between
    Cloudflare, AWS CloudFront/WAF, and Google Cloud CDN when exposed to identical
    attack scenarios?}

  \item \textbf{To what extent do CAPTCHAs and JavaScript challenges reduce automated
    bot traffic without degrading legitimate user access?}

  \item \textbf{Which categories of malicious traffic (e.g.\ brute force, API abuse,
    automated scanning) are most likely to bypass CDN security controls?}

  \item \textbf{What is the rate of false positives (legitimate interactions blocked
    or challenged), and how does this differ across CDN platforms and settings?}

\end{enumerate}

Ultimately, this project aims to shift the conversation around cloud security from
trust-based adoption to evidence-based evaluation. By providing transparent,
reproducible measurements of what actually stops attacks, we enable security
professionals to make informed decisions rather than relying on vendor assurances.

% ---------- Related Work ----------
\section{Related Work}
\label{sec:related-work}

\subsection{Cloudflare CDN Performance: A WordPress Case Study~\cite{cloudflare-wp}}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.8\textwidth]{image2.png}
  \caption{Cloudflare Network Architecture}
  \label{fig:cloudflare-arch}
\end{figure}

This research compared two WordPress websites with similar traffic volumes (25–33k
monthly visitors) hosted on the same provider, with one integrated with Cloudflare
CDN and the other using standard hosting. The Cloudflare-integrated site scored 95\%
on performance metrics compared to 51\% for the non-CDN site, with significant
improvements in time-to-first-byte, start render, and speed index. Memory usage also
differed dramatically: the Cloudflare site peaked at 400\,MB and remained stable,
while the non-CDN site repeatedly hit a 2\,GB memory limit.

\subsection{Hybrid Deep Learning for Cloud Intrusion Detection~\cite{eos-ids}}

An Optimal Security System for intrusion detection in cloud computing environments
(EOS-IDS) was proposed using a three-stage process: (1) data pre-processing via an
Improved Heap Optimisation (IHO) algorithm; (2) feature selection via a Chaotic Red
Deer Optimisation (CRDO) algorithm; and (3) detection and classification via a Deep
Kronecker Neural Network (CKNN). The model successfully addressed common IDS
limitations including over-fitting, low classification accuracy, and high false
positive rates.

\subsection{Intrusion Detection Using Hybrid Deep Learning~\cite{rana2024}}

This MSc research by Kunal Rana investigated ML and deep learning for intrusion
detection using the UNSW-NB15 dataset (nine attack types across 49 features). The
study combined traditional ML (logistic regression, decision trees) with LSTM,
bidirectional LSTM, and autoencoders. Bi-LSTM combined with autoencoders significantly
outperformed traditional methods, emphasising the importance of algorithm selection
and feature extraction strategies.

\subsection{Concept Drift Detection via Pre-Clustering~\cite{concept-drift}}

This work on concept drift detection via pre-clustering and statistical testing
directly informs the drift monitoring approach used in the HDBSCAN pipeline
(Section~\ref{sec:drift}).

\medskip
Taken together, these works highlight two consistent gaps in the literature: the lack
of empirical CDN security evaluation under real attack conditions, and the predominance
of supervised or benchmark-independent approaches to traffic classification. This
project addresses both by combining live honeypot deployment with unsupervised
learning on naturally accumulated traffic data.

\section{Honeypot Design}
\label{sec:honeypot-design}

A legitimate domain (\url{https://hdtvstreams.com}) was purchased and equipped with a
valid SSL/TLS certificate. The certificate significantly increased traffic to the
honeypot: many automated scanning tools and botnets prioritise HTTPS-enabled sites
because they appear more legitimate~\cite{siteguarding} and are more likely to contain
valuable data. The presence of a real domain rather than a bare IP address also
makes the honeypot visible to broader DNS-based bot reconnaissance sweeps.

Within days of deploying the domain and certificate, traffic increased noticeably,
demonstrating how adding security measures (while essential for protecting legitimate
services) paradoxically makes a honeypot more effective by making it indistinguishable
from a real production application.

The honeypot is deployed on a Google Cloud Platform (GCP) virtual machine instance.
To ensure 24/7 availability without requiring an open SSH session, NGINX acts as a
reverse proxy and static file server, while PM2 (a production process manager for
Node.js) keeps the SQLite backend running as a daemon, automatically restarting on
crash.

The application also uses an AWS Lambda function to log interaction events through
the frontend to CloudWatch. When triggered, the Lambda records basic request details
(source IP, request headers, event type) and checks the IP against public reputation
and blacklist sources.

\section{CDN-Fronted Architecture}

All incoming traffic is routed through the CDN before reaching the honeypot. This
allows direct observation of how requests are handled at the CDN layer: blocked
outright, challenged, or forwarded to the backend depending on the active
configuration.

Suspicious requests that reach the application are logged using a serverless pipeline
triggered via an API gateway. Metadata including request paths, headers, IP addresses,
user agents, and timestamps is recorded and stored centrally for later analysis. This
ensures consistent data collection across all test phases. The CDN remains the primary
security control under evaluation, while the honeypot acts as a passive observation
point.

\subsection{Rate Limiting with Locust}
\label{sec:cf-rate-limiting}

\subsubsection{Overview}

To validate Cloudflare's rate-limiting protections, load testing was conducted using
Locust~\cite{locust}, an open-source Python-based performance testing framework chosen
for its developer-friendly, code-based test definitions and gevent-based concurrency,
enabling a single process to simulate thousands of concurrent users.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{image13.png}
  \caption{Locust Load Test Results}
  \label{fig:locust-results}
\end{figure}

\subsubsection{Test Setup}

A single simulated user repeatedly attempted to log into the honeypot's authentication
endpoint (\texttt{/api/auth/login}) using randomly generated credentials from the
Faker library~\cite{faker}. The wait time between requests was set to 10–30\,ms to
simulate an aggressive automated attack. HTTP~429 responses were treated as successful
outcomes (rate limiter functioning correctly), and the script captured and logged the
attacking IP (\texttt{140.203.12.65}) and the exact request count before each trigger.

\subsubsection{Results}

The Cloudflare rate-limiting rule was configured to trigger after 100 requests from a
single IP within a 10-second window, blocking that IP for 10 seconds (the maximum
available on free tier). The first trigger occurred at approximately 108 requests,
validating the threshold.

\subsubsection{Security Implications}
\label{sec:cf-rate-implications}

While the initial 100-request burst was blocked, the same IP could immediately
resume attack cycles after each 10-second timeout. Over 24 hours, this theoretically
permits up to 864,000 credential attempts (100 requests every 10 seconds = 36,000 per
hour). Cloudflare does not natively support permanent or escalating blocks based on
repeated violations; custom solutions using the Cloudflare API are required for
persistent blocking~\cite{cf-community-blocking}.

\subsubsection{Potential Solutions}

\begin{enumerate}
  \item Increase the maximum block duration (up to 1 hour via WAF rules interface) to
        reduce daily attempts from hundreds of thousands to at most 2,400.
  \item Implement progressive penalties: trigger a 1-hour block after five consecutive
        violations of the 10-second window rule.
  \item Build a custom solution using Cloudflare's API in conjunction with a Lambda
        function to permanently block IPs that repeatedly exceed the threshold.
\end{enumerate}

\subsubsection{TCP SYN Flood (Layer 4)}

\paragraph{Overview}

A TCP SYN flood exploits the TCP three-way handshake by sending a high volume of SYN
packets with spoofed source IPs, resulting in half-open connections. The server waits
for final ACKs that never arrive, exhausting its connection table and preventing
legitimate connections~\cite{cf-syn-flood}.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.5\textwidth]{image20.png}
  \caption{TCP SYN Flood Exploitation}
  \label{fig:syn-exploit}
\end{figure}

\paragraph{Attack Setup}

\begin{itemize}
  \item \textbf{Target:} Honeypot on GCP, fronted by Cloudflare.
  \item \textbf{Tool:} hping3 on Kali Linux VM.
  \item \textbf{Monitoring:} Nginx server logs, Cloudflare real-time analytics, GCP
        observability dashboard.
\end{itemize}

\begin{lstlisting}[language=bash, caption={hping3 TCP SYN Flood Commands}]
sudo hping3 -S --flood -p 80 --rand-source <target-IP>
sudo hping3 -S -i u100 -p 80 hdtvstreams.com
\end{lstlisting}

\paragraph{Results}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\textwidth]{image22.png}
  \caption{Resource Utilisation During SYN Flood}
  \label{fig:syn-resources}
\end{figure}

CPU utilisation remained unaffected; the regular spikes visible in the dashboard are
due to pre-existing scripts. Network traffic showed a minor increase, peaking at only
17.21\,KiB/s at the origin — negligible for a production-grade service. SYN-ACK
responses were initially returned from Cloudflare's edge IP (\texttt{172.67.213.165}),
confirming the handshake was handled at the edge, not the origin. As the flood
intensified, intermittent ``ICMP Network Unreachable'' messages appeared, suggesting
additional filtering was triggered.

  \label{fig:syn-icmp}
\end{figure}

\paragraph{Analysis}

Cloudflare absorbed and mitigated the SYN flood before it could affect the origin
server. The stable CPU and negligible network utilisation at the origin confirm
effective edge mitigation. However, the lack of telemetry on the free tier makes it
difficult to diagnose root causes or understand the exact mitigation path.

\subsubsection{UDP Flood (Layer 4)}

\paragraph{Overview}

UDP is a connectionless ``fire-and-forget'' protocol: unlike TCP, no acknowledgement
packets are involved. A UDP flood sends massive packet volumes without maintaining
connection state, targeting random ports and forcing the server to generate ICMP
``Destination Unreachable'' responses for each, consuming bandwidth and CPU.

\paragraph{Attack Setup}

\begin{itemize}
  \item \textbf{Target:} Honeypot on GCP, fronted by Cloudflare.
  \item \textbf{Tool:} hping3 on Kali Linux VM.
\end{itemize}

\begin{lstlisting}[language=bash, caption={hping3 UDP Flood Commands}]
sudo hping3 -2 --flood -p 80 --rand-source <target-IP>
sudo hping3 -2 -i u100 --spoof -p 80 hdtvstreams.com
sudo hping3 -2 -i u100 -p 80 hdtvstreams.com
\end{lstlisting}

Notably, during one run the attacking machine was itself rate-limited for

\paragraph{Analysis}

Silent edge-level drops mean network-layer UDP floods never generate dashboard events
on free-tier plans. While the protection functioned correctly, the complete absence of
logging prevents post-incident analysis.

HTTP connections are opened but data is sent intentionally at low rates, holding
connections open and exhausting the server's connection pool. SlowHTTPTest supports
Slowloris (slow headers), Slow HTTP POST (slow body), and slow-read attack modes.

\subsection{SQL Injection Testing — Cloudflare Free Tier}
\label{sec:cf-sqli}

\subsubsection{Overview}

SQL injection remains one of the most prevalent and dangerous web application
vulnerabilities, consistently appearing in the OWASP Top 10~\cite{owasp-waf}.
This test evaluated whether Cloudflare's free-tier WAF managed rules would detect
and block automated SQL injection tooling against the honeypot's login endpoint
(\texttt{/login} and \texttt{/api/auth/login}), using both the industry-standard
SQLMap scanner and the WAF-bypass-focused Ghauri framework. All tests were conducted
from a Kali Linux virtual machine.

\begin{itemize}
  \item \textbf{Attack OS:} Kali Linux
  \item \textbf{Tools:} SQLMap (v1.10.2) and Ghauri (v1.4.3)
  \item \textbf{Interception:} Burp Suite Community Edition
  \item \textbf{Target:} \url{https://hdtvstreams.com} via Cloudflare Free Tier Edge
\end{itemize}

% ----- Test 1 -----
\subsubsection{Test 1 — Automated Form Discovery}

\begin{lstlisting}[language=bash, caption={SQLMap Form Discovery}]
sqlmap -u https://hdtvstreams.com/login --forms --batch
\end{lstlisting}

\paragraph{Result}

SQLMap returned HTTP~405 (Method Not Allowed) responses throughout the scan. Without
a \texttt{--data} flag, the tool had no POST body to target and defaulted to probing
the \texttt{Referer} and \texttt{Host} headers instead of the intended form fields.
No injection point was identified. More significantly, \textbf{no Cloudflare WAF
event was triggered} and no block page was returned — the 405 response originated
entirely from the origin server's Nginx configuration, not from any CDN-layer
filtering. The SQLMap default user-agent passed through the Cloudflare edge without
challenge.

% ----- Test 2 -----
\subsubsection{Test 2 — Aggressive Payload Injection}

\begin{lstlisting}[language=bash, caption={SQLMap Aggressive Test — Maximum Risk and Level}]
sqlmap -u https://hdtvstreams.com/login \
  --data="email=test@example.com&password=test" \
  --risk 5 --level 5 \
  --batch
\end{lstlisting}

\paragraph{Result}

With \texttt{--data} supplied, SQLMap correctly targeted the \texttt{email} and
\texttt{password} POST body parameters. The \texttt{--level 5} flag additionally
enabled header injection testing across \texttt{User-Agent}, \texttt{Referer}, and
\texttt{Host}. SQLMap reported both POST parameters as ``not dynamic'' — the server
returned an identical response body regardless of payload content — depriving the
tool of the differential signal required for detection. The scan concluded with no
injectable parameters found.

Critically, \textbf{zero Cloudflare WAF security events were generated} across the
entire run. Risk~5 / Level~5 is the most exhaustive SQLMap configuration, producing
a high volume of highly characteristic SQL injection payloads in the POST body;
none were flagged, blocked, or challenged by Cloudflare's managed ruleset.

% ----- Test 3 -----
\subsubsection{Test 3 — Evasion with Tamper Scripts and Random User Agent}

\begin{lstlisting}[language=bash, caption={SQLMap with Tamper Scripts and Random Browser UA}]
sqlmap -u https://hdtvstreams.com/login \
  --data="email=test@example.com&password=test" \
  --tamper=randomcase,space2comment,charencode \
  --random-agent \
  --batch
\end{lstlisting}

\paragraph{Result}

This test introduced two evasion techniques simultaneously: tamper scripts to
obfuscate payloads (case randomisation, comment-space substitution, and URL character
encoding), and \texttt{--random-agent} to replace SQLMap's identifiable default
user-agent with a randomised real browser string.

The \texttt{--random-agent} flag produced the most significant finding of the entire
Cloudflare SQL injection test series. With a browser-like user agent, all requests
bypassed the Cloudflare edge without any challenge and hit the origin server directly.
The Cloudflare traffic dashboard recorded \textbf{59,950 requests originating from a
single Irish IP address} (ASN~1213 — HEAnet, \texttt{140.203.12.65}) with
\textbf{zero mitigation actions applied}. Every request detail entry showed
``Suspicious activity: None'' and ``Mitigation: Not mitigated''
(Figures~\ref{fig:cf-sqli-dashboard} and~\ref{fig:cf-sqli-detail}).

% ----- Test 4 -----
\subsubsection{Test 4 — Ghauri WAF-Bypass Framework}

\begin{lstlisting}[language=bash, caption={Ghauri with POST Request File and Random UA}]
ghauri -r post.txt --random-agent --batch --ignore-code 401
\end{lstlisting}

\paragraph{Result}

Ghauri initially returned 401 Unauthorized responses; adding \texttt{--ignore-code 401}
instructed the tool to proceed regardless. Ghauri tested a comprehensive range of
injection techniques — error-based, boolean-based blind, time-based blind, and
stacked queries — across both \texttt{email} and \texttt{password} fields. The run
concluded with:

\begin{lstlisting}
[CRITICAL] all tested parameters do not appear to be injectable.
\end{lstlisting}

As with all SQLMap runs, \textbf{no Cloudflare WAF rule fired} and no security event
appeared in the dashboard. Ghauri is specifically designed to bypass WAF fingerprinting;
its evasion techniques proved unnecessary given that Cloudflare's free tier generated
no detectable response to any of the injected payloads.

\subsubsection{Analysis}

Across all four test runs — spanning basic form discovery, maximum-aggression payload
injection (\texttt{--risk 5 --level 5}), multi-technique tamper-script evasion, and a
dedicated WAF-bypass framework — Cloudflare's free-tier WAF generated
\textbf{zero security events} and applied \textbf{zero mitigations}.

Two independent factors explain why no successful injection occurred despite the
complete absence of WAF intervention:

\begin{enumerate}
  \item \textbf{Parameterised queries.} The Node.js backend uses SQLite with
        parameterised statements (\texttt{db.get("SELECT * FROM users WHERE email = ?
        AND password = ?", [email, password])}), which are structurally immune to SQL
        injection regardless of WAF coverage.
  \item \textbf{Uniform server responses.} The login endpoint returned an identical
        response body for all inputs, preventing injection tools from obtaining the
        differential signal required for blind injection detection.
\end{enumerate}

The critical security implication is that an identical application using string-concatenated
queries would have been \textbf{fully exposed}: Cloudflare's free-tier managed rules
provided no payload-level detection across any of the four scenarios. A simple
\texttt{--random-agent} flag was sufficient to ensure 59,950 requests reached the
origin with no challenge applied. This represents a meaningful gap in the free-tier
WAF's out-of-the-box SQL injection coverage, with the application's own parameterised
queries acting as the sole effective defence.

\subsubsection{TCP SYN Flood and UDP Flood (Layer 4)}

Both network-layer flood attacks were conducted using hping3 with the same commands
used against Cloudflare, allowing a direct side-by-side comparison.

\paragraph{TCP SYN Flood}
A SYN flood exploits the TCP three-way handshake by sending a high volume of SYN
packets, exhausting the target's connection table with half-open connections that
never complete.

\begin{lstlisting}[language=bash, caption={hping3 TCP SYN Flood Commands}]
sudo hping3 -S --flood -p 80 --rand-source <target-IP>
sudo hping3 -S -i u100 -p 80 hdtvstreams.com
\end{lstlisting}

\paragraph{UDP Flood}
UDP is connectionless and stateless; a flood sends massive packet volumes to random
ports, forcing the server to generate ICMP ``Destination Unreachable'' responses for
each, consuming bandwidth and CPU.

\begin{lstlisting}[language=bash, caption={hping3 UDP Flood Commands}]
sudo hping3 -2 --flood -p 80 --rand-source <target-IP>
sudo hping3 -2 -i u100 --spoof -p 80 hdtvstreams.com
sudo hping3 -2 -i u100 -p 80 hdtvstreams.com
\end{lstlisting}

\paragraph{Results}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\textwidth]{image63.png}
  \caption{hping3 Logs During SYN and UDP Flood Testing — 100\% Packet Loss}
  \label{fig:bunny-hping-logs}
\end{figure}

Both attacks produced identical results across every run:

\begin{itemize}
  \item \textbf{100\% packet loss} — not a single SYN or UDP packet received a
        response from either the edge or the origin.
  \item \textbf{Zero CPU and network impact} at the origin — the GCP instance
        remained completely unaffected throughout both tests.
  \item At points during both runs, hping3 was \textbf{unable to resolve the target
        host entirely}, suggesting BunnyCDN applied temporary IP-level blocking on top
        of packet filtering — a more aggressive response than silent dropping alone.
\end{itemize}

\paragraph{Analysis}

BunnyCDN absorbed both Layer 4 floods entirely at the edge, with no traffic reaching
the origin server. Unlike Cloudflare, which returned SYN-ACK responses before
beginning to filter, BunnyCDN dropped all packets silently from the first request —
indicating filtering occurs upstream of the TCP handshake. The temporary host
resolution failures observed during both floods suggest an additional IP-level
throttling mechanism beyond standard packet filtering. However, as with Cloudflare's
handling of network-layer attacks, \textbf{neither the SYN flood nor the UDP flood
generated any dashboard events} — silent mitigation at this layer means
administrators would have no visibility into the attack without external monitoring.

\chapter{Traffic Pattern Analysis}

\section{Overview}

The aim of this phase is to analyse live honeypot traffic in five-minute windows,
using these request buckets to group similar request patterns and generate actionable
security insights. By clustering logs, high-volume bot behaviour and targeted
malicious activity can be identified to prevent exploitation of potential
vulnerabilities. An unsupervised clustering approach works in the right ``adaptive''
direction: it can highlight new anomalies as they arise and provide references to
similar previous attacks, speeding up attack identification and surfacing new
malicious behaviour.

\section{Clustering Approach}

\subsection{Algorithm 1: K-Means Clustering}

\subsubsection{Overview}

K-Means is a popular unsupervised machine learning algorithm that partitions a dataset
into a pre-specified $k$ clusters, each defined by a centroid (the average position of
all its members). The algorithm iterates: randomly initialise $k$ centroids, assign
each point to the nearest centroid, recalculate centroids, and repeat until convergence.

\subsubsection{Pre-processing}

Client IPs and timestamps were extracted from raw honeypot logs, then traffic was
aggregated into 5-minute time windows per IP. Features included request counts,
unique path counts, status code distributions (401, 403, 404, 200), and TF-IDF
vectors of request sequences. Numerical features were standardised using scikit-learn's
\texttt{StandardScaler}~\cite{sklearn-scaler}; TF-IDF vectors were horizontally
stacked. The dataset was split 70:30 (431 training / 185 test samples).

\subsubsection{Model Training}

The elbow method was used to determine optimal $k$ (tested 2–10 clusters). The
optimal value was identified as $k=5$.

\subsubsection{Limitations}

K-Means creates spherical clusters with hard boundaries, forcing every point into a
cluster regardless of whether it represents genuine attack behaviour or noise.
The PCA visualisation (Figure~\ref{fig:kmeans-pca}) highlights fragmentation: points
assigned to the same cluster occupy widely separated regions of the feature space.
These limitations prompted a transition to HDBSCAN.

% ============================================================
\subsection{Algorithm 2: HDBSCAN Clustering}

\subsubsection{Overview}

HDBSCAN~\cite{sklearn-hdbscan} (Hierarchical Density-Based Spatial Clustering of
Applications with Noise) addresses K-Means' limitations: it discovers clusters based
on density variations, automatically determines the number of clusters, identifies
noise points (labelled $-1$), and handles clusters of varying shapes and densities.

The algorithm: (1) calculates core distances based on the $k$-th nearest neighbour;
(2) constructs a mutual-reachability distance graph; (3) builds a minimum spanning
tree; (4) converts it into a hierarchy of connected components (dendrogram); (5)
extracts stable clusters using the Excess of Mass method.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.7\textwidth]{image93.png}
  \caption{Dendrogram Produced During HDBSCAN Hierarchical Clustering}
  \label{fig:dendrogram}
\end{figure}

\subsubsection{Advantages for Security Applications}

\begin{itemize}
  \item Explicitly identifies noise (label $-1$), distinguishing genuine attack
        patterns from random benign traffic.
  \item Handles clusters of varying densities — essential when different attack types
        exhibit vastly different activity levels.
  \item Multi-scale pattern detection via the cluster hierarchy captures reconnaissance
        followed by escalated exploitation as related patterns.
  \item Parameters (\texttt{min\_cluster\_size}, \texttt{min\_samples}) have direct
        security interpretations: minimum attack footprint worth investigating and
        sensitivity to isolated suspicious events.
\end{itemize}

\section{Job Scheduling and Monitoring}

Initially, pipeline jobs were triggered using standard cron jobs executed directly
from the instance terminal. While functional, this approach offered no visibility
into run history, no alerting on failure, and no resource tracking. To improve
monitoring and reliability, scheduling was migrated to
Cronicle~\cite{cronicle} --- a web-based job scheduler and monitoring system.

Cronicle provides a visual interface showing the status of each job run with clear
indications of successful completions and failures. The platform sends notifications
when jobs fail, enabling quicker troubleshooting and resolution if something goes
wrong. It also tracks performance metrics per execution --- CPU usage, memory
consumption, and wall-clock execution time --- which have been valuable in confirming
the pipeline does not approach the VM's resource limits. Based on the metrics
collected across hundreds of runs, it is far from doing so: each run averages 2
seconds and consumes only 0.5\% of CPU.

\section{Agentic LLM Interpretability}

\subsection{Overview}

While the clustering model assigns traffic windows to behavioural clusters and scores
confidence, a label like ``IoT / embedded device exploit scan'' with a confidence of
0.023 provides little actionable guidance. The LLM layer does not replace the
clustering output; it interprets and enriches it into something actionable.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{image103.png}
  \caption{Agentic LLM Pipeline Architecture}
  \label{fig:llm-arch}
\end{figure}

\subsection{Context Supplied to the Model}

On each pipeline run the model receives:

\begin{itemize}
  \item The current rollup and prediction rows.
  \item Cluster severity labels.
  \item Active Cloudflare WAF rules (as an example context).
\end{itemize}

From this it produces a human-readable threat summary, specific WAF actions tied to
each detected cluster, and a triage priority list ordered by severity. For every
Medium, High, or Critical cluster it raises a structured Jira ticket containing the
observed activity, matched patterns, recommended action, and supporting references.
The model is explicitly instructed not to fabricate CVE URLs to prevent
hallucinations.

\subsection{RAG and Knowledgebase Integration}

The MITRE D3FEND~\cite{d3fend} knowledgebase (thousands of entries) cannot be
pasted wholesale into the prompt. Instead, each entry is pre-embedded offline using
\texttt{sentence-transformers/all-MiniLM-L6-v2} via the HuggingFace Inference API
and stored as normalised vectors. At query time, the model calls a
\texttt{query\_knowledgebase} tool, and cosine-similarity retrieval injects the three
most relevant entries into the prompt, keeping the context window lean while grounding
recommendations in established frameworks.

\subsection{Agentic Tool-Use Loop}

The model operates as a tool-use loop, making sequential calls within a single run:

\begin{enumerate}
  \item Query the knowledgebase once per detected cluster.
  \item Call \texttt{check\_existing\_jira\_tickets} to avoid duplicate tickets.
  \item Create a new ticket or append an update to an existing one.
\end{enumerate}

Cluster 13 (known legitimate application traffic, labelled ``Legitimate application
traffic'' with confidence 0.598) is suppressed from ticket creation
unless anomalous signals appear.
